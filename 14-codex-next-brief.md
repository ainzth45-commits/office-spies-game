## A. 🐛 แก้ลอจิก flow (สำคัญ — unit test จับไม่เจอ แต่กดจริงพัง)

### A1. หน้าประกาศผลโหวต + §4.5 ถูก "ข้าม" ไม่ได้แสดง 🔴
- **ปัญหา:** `finalizeVoteRound()` (state/actions.ts) เซ็ต `phase` เป็น `guess`/`refund`/`postVoteClue` **ทันที** → พอกด "คำนวณและประกาศผล" ใน `VoteResultScene` แล้ว phase เด้งออกจาก `voteResult` ทันที → **บล็อกแสดงผลในบรรทัด 29–46 (ชื่อผล "จับสปายได้!/จับผิดคน!/โหวตพลาด!" + §4.5 "มีสายลับ X จาก Y") ไม่ถูก render เลย** กระโดดข้ามไปฉากถัดไป
- **แก้:** ให้ `finalizeVoteRound()` เซ็ต `phase: "voteResult"` (คงไว้) — คำนวณ result/refund/consume shield เหมือนเดิม แต่ไม่เปลี่ยน phase
  - `VoteResultScene` มีปุ่ม "ไปต่อ" ที่ route ไป `guess`/`refund`/`postVoteClue` อยู่แล้ว (บรรทัด 40–44) → ใช้ตัวนั้นเป็นตัวเดินหน้าแทน
- **เทส:** หลัง `finalizeVoteRound` → `phase==="voteResult"` และ `lastVoteResult` ถูกตั้ง

### A2. กรณี "จับสปายได้" ข้าม §4.6 ซื้อเบาะแส 🟠
- **ปัญหา:** caughtSpy → ไป `guess` ตรงๆ ข้ามช่วงซื้อเบาะแส แต่สเปก §4.5/§4.6 บอกว่าทำงาน **หลังประกาศผลทุกกรณี (แพ้และชนะ)**
- **แก้ (แนะนำ):** caughtSpy → `voteResult` (โชว์ §4.5) → "ไปต่อ" → `postVoteClue` (ทีมเลือกซื้อเบาะแส 1 ครั้ง หรือข้าม) → แล้วมีปุ่ม **"ทายสายลับคนที่สอง"** เข้า `guess`
  - ให้เบาะแสช่วยทีมก่อนทาย (ตรงเจตนาเกม)
- เช็คให้ `PostVoteClueScene` มีปุ่มเข้า `guess` เมื่อผลคือ caughtSpy

### A3. เกราะ "เกิดใหม่ได้หลังถูกใช้" — ผิดสเปก 🟠
- **ปัญหา:** `applyGachaOutcome` ใช้ `shieldAvailable = !state.shield.exists || state.shield.consumed` → พอเกราะถูกใช้ (consumed=true) จะ **สุ่มได้เกราะใหม่อีก** แต่สเปก §6.5 ว่า "มีชิ้นเดียวในเกม ถูกใช้แล้วหายจากเกม"
- **แก้:** เปลี่ยนเป็น `shieldAvailable = !state.shield.exists` (เกราะออกได้ครั้งเดียวต่อเกม — ใช้แล้วไม่เกิดอีก) + เทสยืนยัน
- *(ถ้าเจ้านายอยากให้เกิดใหม่ได้จริง บอก codex คงไว้ตามเดิม — แต่ default ควรตรงสเปก)*

### ✅ ส่วน flow ที่ตรวจแล้ว "ถูกต้อง" (คงไว้)
- vote engine, R/P, เกราะ silent flip, spy-pool, clueEngine — ตรงสเปก + เทสผ่าน
- `submitVoteTurn` (กันโหวตซ้ำ/ไม่อยู่/หักไอเทมถูก), `buyVoteItem` (ลิมิตกระเป๋า+รายวัน), `applyGachaOutcome` (10 ผล + fallback + multiplier + grantItem/grantQuiz/shield), `resolveSecondSpyGuess` (re-randomize คง inventory/shield), `markFinalDay` guard — ถูกต้อง ✓

## B. 🎨 ภาพ & ฟิลเกม (Task 10 — งานหลักที่เหลือ)

ตอนนี้เป็น UI สี+ฟอนต์ล้วน **ยังไม่มีภาพ/เสียง/อนิเมชันเลย** → ยังไม่รู้สึกน่าเล่น

**ลำดับที่กระทบความรู้สึกมากสุด → ทำก่อน:**
1. **โลโก้ + ฉากหลัง** — `logo`, `bg-home-case-board` (Home), `bg-role-cover` (ม่านส่งเครื่อง), `bg-reveal` (ฉากผล)
2. **การ์ดบทบาท** — `role-normal`, `role-spy` (+ ออร่าแดงตอนเป็นสปาย), badge คู่หู
3. **กาชา** — `gacha-machine` + อนิเมชันสั่น→แคปซูลหล่น→เปิดป๊อป + confetti (ตอนนี้เป็นแค่ "?")
4. **ตราผลโหวต** — `stamp-win`/`stamp-lose` กระแทกลงจอ + กลองรัว (ใช้ใน VoteResultScene)

## C. ✍️ คำอธิบาย / UX writing (ปรับเยอะ)

ข้อความตอนนี้ "ใช้ได้แต่แห้ง + มีร่องรอย dev" ผู้เล่นเรียนเกมที่ซับซ้อนสดหน้างาน → **ทุกหน้าต้องบอกชัดว่า "ตอนนี้ทำอะไร / กดอะไรต่อ"** ด้วยภาษาสนุกเป็นมิตร

**หลักการ:**
- ทุกหน้ามี 1 บรรทัดบอก "ทำอะไรตอนนี้" + ปุ่มบอกผลลัพธ์ชัด
- ลบข้อความสาย dev ทิ้ง (เช่น "รอบนี้ยังเป็น vote shell")
- โทนขี้เล่นแบบนักสืบ แต่กติกาเป๊ะ
- ห้ามหลุดความลับในข้อความ (เกราะ/บทบาท/จำนวนเสียง)

**จุดที่ต้องเขียนใหม่ (ตัวอย่าง):**
- `VoteResultScene` — "ห้ามดูตัวเลขเสียง ระบบจะประกาศเฉพาะผลรวม" → เขียนใหม่ให้ลุ้น เช่น *"ปิดปากเงียบ! ระบบกำลังนับ... เปิดผลได้เลย"*; กรณี failed อธิบายเป็นมิตร *"ครั้งนี้ยังจับไม่ได้ เหรียญที่ลงไปคืนซุง — ลองใหม่วันหลังนะ"*
- ทุกหน้า private (role/ซื้อไอเทม/โหวต) — เพิ่มบรรทัดนำ "เดินมาที่เครื่องทีละคน • คนอื่นห้ามแอบดู"
- ม่านส่งเครื่อง — ข้อความชัดว่า "ส่ง iPad ให้คนถัดไป"
- ไอเทมแต่ละชิ้น — คำอธิบายสั้นว่าใช้ทำอะไร (โดยเฉพาะ R/P ที่ซับซ้อน)
- §4.5 / เบาะแส — เขียนให้เข้าใจว่าเอาไปทำอะไรต่อ ("เอาไปถามกันว่าใครโหวตใคร")
- หน้า Settings — label + คำอธิบายสั้นของแต่ละค่า (กันแอดมินตั้งพลาด)
- เพิ่มปุ่ม **"กฎย่อ"** เปิดดูกติกาเร็วๆ ได้ทุกเมื่อ (เสริมจาก tutorial)
